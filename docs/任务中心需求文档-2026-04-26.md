## 1. 用户原始需求摘录

> 以下为用户在 brainstorm 起点的完整原文（已分段标号），后续 brainstorm 围绕这段展开：

### 1.1 起点

> "我想做一个任务中心，发起点是明空选品里面的产品和视频素材，管理员通过从里面发现适合搬运到小语种国家的视频素材，选择商品，选择素材之后，形成一条新的素材管理里面的条目，如果商品本身已经在素材管理库，则是新增一条原始英文素材。"

### 1.2 原始素材任务库（D）

> "已经在库里的产品，那么它会在原有的素材管理库中新增一条英文视频。但由于它目前没有原始视频，我们需要新增一个功能：
> 1. 建立原始素材任务库：我们需要新增一个名为'原始素材任务库'的菜单。
> 2. 任务流转与预处理：(a) 所有选中的新原始英文素材进入素材管理库后，如果没有对应的原始视频，该条目会自动进入'原始视频任务库'。(b) 相关人员可以从任务库中认领并下载视频。(c) 完成预处理（例如去字幕、去尾巴等操作）后，再进行上传。
> 3. 自动绑定：上传后的视频会自动提交并绑定到素材库中对应条目的'原始素材'按钮下。这样，视频与这条原始英文素材的绑定关系就正式建立了。"

### 1.3 新品审核（B）

> "关于新品审核与评估流程的设计需求如下：
>
> 1. 业务逻辑区分：老品处理：若选中的产品为老品，直接沿用后续流程，说明产品本身已验证无误。新品定义：若产品在'素材管理库'中从未出现过、且无任何绑定关系，则定义为'新品'。
>
> 2. 页面与数据库设计：功能入口：在菜单栏左侧'明控选品'逻辑中，在现有页面增加两个 Tab：1. 明控选品；2. 新品审核。数据库支持：需对应一张独立的新品数据库表。该表负责接收管理员在'明控选品'中选中的新品及其关联素材。存储字段：商品链接 / 商品英文标题 / 商品主图 / 对应的视频素材。
>
> 3. 素材预处理：使用 FFmpeg 将视频素材统一截断至 15 秒左右。生成专门用于 AI 评估的短视频片段。
>
> 4. AI 评估逻辑：进入'新品审核'评估环节后，调用 AI 大模型（首选 Gemini 1.5 Pro，备选 GPT-5.5 等）。输入参数：产品主图、标题、链接及预处理后的带货短视频。评估原则：针对单一市场进行精准评估（不建议九国市场汇总评估，以保证准确性）。评估输出：判断该产品是否适合在已启用的各小语种国家推广。
>
> 5. 前端交互与展示：评估矩阵：页面需展示国家维度（假设 9 个国家）的评估列，结果以'勾选（对号）'或'打叉'表示。详情查看：悬浮显示：鼠标悬停在图标上时，即时加载显示 AI 给出的具体评分及原因。弹窗显示：点击图标后弹出窗口，展示详细的评估逻辑（可参考现有视频素材库中具备条件的 AI 评估写法）。
>
> 6. 任务分发与操作：管理员可对已有 AI 评估结果的产品进行后续操作：国家筛选：点击'上架'或'不上架'时弹出国家勾选框。弹窗内需预先回显 AI 的建议结果（推荐上架的国家及原因）。交互按钮：每一行下方设有确认按钮。默认为'不上架'（底色）。点击后变为'上架'（蓝色）。负责人分配：完成勾选后，需从现有非管理员员工账号中选择一名负责人。流程闭环：若为新品/新素材，完成上述步骤即完成任务分发。若为老品选用了新素材，负责人自动沿用原有的负责人。"

### 1.4 任务中心（C）+ 翻译任务池（E）+ 报表（F）

> "我核心的逻辑是想在选品之后，建立一个中间的'任务中心'逻辑。通过这种方式形成一条新的任务流，具体设想如下：
>
> 1. 界面与权限：(a) 在左侧增加一个'任务中心'的大菜单。(b) 不同角色的权限不同：普通员工进去后，只能看到分配给自己的任务。(c) 任务中心页面在布局上可以参考素材管理页面，但在功能上要支持原有的翻译提交能力（如翻译封面、视频等）。
>
> 2. 任务流转闭环：(a) 视频任务生成并通过审核后，不应直接流向剪辑翻译人员。(b) 第一步：任务先流转给负责处理原始视频的人员。(c) 第二步：原始视频处理好并上传后，任务才进入普通翻译员工的任务池。(d) 第三步：翻译人员基于任务完成封面、视频、文案的翻译，并处理好相关链接。(e) 第四步：任务标注为'完成'，整条素材状态变为'待推送'。
>
> 3. 系统化管理：(a) 核心目的：将整个任务流转闭环掉，让每个人都知道去哪里领任务，而不是像现在这样靠口头交代，通过系统化来解决问题。(b) 任务提交后进入推送管理，之后再进入投放环节。后续我们还可以考虑如何接入投放数据。(c) 流程中应包含'打回'机制，并有相应的审核环节。
>
> 4. 效率与考核：(a) 系统需要统计员工每天完成的任务量。(b) 记录任务的质量数据：有多少是完全通过的，有多少需要返工。(c) 通过这些数据进行通过率考核，从而精准掌握每个员工每天的工作状态和效率。
>
> 业务流转逻辑：(a) 初始启发点在'名空选品'端。(b) 在选品端选定商品后，需要将其存入'素材管理池'，接着进入'任务池'。(c) 针对该商品及其关联素材的整套逻辑必须完全打通。"

### 1.5 实施节奏

> "关于这个任务中心，我们可以先采取以下步骤：1. 先通过手工维护。2. 等对应的程序流程跑通之后，再实现全线打通，即从'名空选品'那边直接生成数据到这边。这样处理的话，本质上只是从'手工填写输入'转变为'由上游数据源自动输入'。"

## 3. 已锁定决定清单（贯穿所有子系统）

> 完整版见 [master 文件第 3 节](superpowers/specs/2026-04-26-mingkong-pipeline-master.md#3-已锁定决定清单贯穿子系统)。
>
> **核心 14 条**（不重复全文，仅列编号 + 一句话）：

### 任务模型与状态
1. 双层任务模型（父=素材级 / 子=国家级）
2. 两道审核（原始视频 + 翻译提交）
3. 子任务在父任务创建时一并物化为 `blocked`，原始素材审核通过批量解锁
4. 父任务原始视频处理人**不必填**，进认领池
5. 一人一品（产品级 owner，子任务全部给同一翻译员）
6. 老品自动沿用 `media_products.user_id`，新品 admin 选
7. 打回 = 原路返回；任务中心不带换人按钮；换人在素材管理做
8. owner 变更级联：状态非 done/cancelled 的子任务跟换；已完成 / 已取消冻结快照
9. 取消（cancel）是终态：父取消级联非 done 子任务，已 done 保留

### 权限与角色
10. 角色保持现有（superadmin/admin/user），不加新 role
11. `users.permissions` JSON 加 `task_center` / `can_process_raw_video` / `can_translate`
12. 一人可同时拥有两种能力位

### 集成与桥接
13. 子任务页半集成：复用素材管理"翻译"+ "翻译任务记录"按钮，输入受任务范围锁定
14. 父任务"上传"= 跳素材管理上传，回任务中心标 tag（D 阶段强化）
15. "完成"按钮 = 后端调 `appcore.pushes.compute_readiness` gate 检查
16. "待推送"是 `pushes_list` 实时算的，无需任务中心写额外标志
17. 国家代码大写 ISO（DE / FR / JA / NL / SV / FI / ...）
18. 审计写 `task_events` 表，event_type 枚举完整覆盖所有状态翻转

### UI / UX
19. Ocean Blue 设计系统（参考根 CLAUDE.md）
20. 任务中心默认页"待派单素材"Tab + 一键创建任务
21. 同时拥有两能力位的用户在"我的任务"看到混合（父+子）
22. 打回 / 取消 reason 必填 ≥10 字符
23. admin 自审（actor==assignee）不拦

### LLM 调用
24. 走 `appcore.llm_client`，B 子系统在 `llm_use_cases.py` 加 use_case，不直接 `from openai import`
