# 多语言标题文案翻译设计

**日期**: 2026-04-17  
**状态**: 待评审  
**触达模块**: `web/` 新增独立菜单与工作台页，`appcore/` 或 `pipeline/` 新增语种 prompt 配置与翻译执行逻辑，`tests/` 新增路由与校验测试

---

## 1. 背景

当前项目里已经有通用型的 `text_translate` 文本翻译工作台，但它的定位是“通用文本翻译”：

- 支持自由文本和分段输入
- 支持源语言/目标语言双向切换
- 支持手动编辑 prompt
- 支持切换 provider

这与本次需求不一致。本次要做的是一个专门面向电商短视频文案的“小语种标题文案翻译”工具，特点非常明确：

- 页面是独立菜单，不复用现有通用翻译页的交互
- 输入源文固定为英文
- 输入结构固定为三段：`标题`、`文案`、`描述`
- 顶部目标语种动态来自 `media_languages.enabled=1`
- 用户每次只翻译一个语种
- 调用模型固定为 Sonnet，不开放 provider 切换
- 语种 prompt 由后端内置，不提供页面持久化编辑

如果继续在 `text_translate` 上打补丁，会把“通用翻译”和“专用营销文案翻译”两套模型混在一起，后续维护成本会越来越高。因此本次采用独立菜单和独立接口。

---

## 2. 目标

1. 新增一个独立菜单页，用于“多语言标题文案翻译”。
2. 顶部语种胶囊从 `media_languages` 表动态读取，只显示 `enabled=1` 的目标语种。
3. 输入区只接受固定三段结构：
   - `标题: ...`
   - `文案: ...`
   - `描述: ...`
4. 输入格式不合法时，前端直接报错，不发请求；后端也做同样的严格校验。
5. 点击翻译后，后端组合“当前语种默认 prompt + 用户输入原文”，调用 Sonnet 完成翻译。
6. 结果区固定展示三段翻译结果：标题、文案、描述。
7. 第一版不做历史记录，不创建 `projects` 任务，不做持久化翻译记录，只提供单页即时翻译能力。

---

## 3. 非目标

- 不复用 `text_translate` 的项目列表、详情页和历史记录模型
- 不支持任意自由格式英文文本
- 不支持多语种同时翻译
- 不支持页面上编辑并保存各语种 prompt
- 不支持 provider 切换或模型切换
- 不做批量导入、批量导出、翻译队列或任务中心

---

## 4. 方案比较

### 方案 A：复用 `text_translate`

做法：

- 保留现有 `/text-translate`
- 移除或隐藏多余能力
- 增加语种胶囊和固定格式校验

优点：

- 开发量最小

缺点：

- 现有页面语义是通用翻译，不是专用文案翻译
- 多段输入、provider 选择、prompt 编辑等能力会继续污染页面
- 未来继续扩展该功能时容易和通用翻译逻辑缠在一起

### 方案 B：独立菜单页 + 语种 prompt 全部硬编码

做法：

- 新增独立菜单和页面
- 每种语种的默认 prompt 直接写死在路由或配置模块里

优点：

- 边界清晰
- 第一版实现简单

缺点：

- 后台新增启用语种后，如果代码里没有对应 prompt，就会不可用
- 动态语种和静态 prompt 的扩展节奏不一致

### 方案 C：独立菜单页 + 动态语种 + 内置精调 prompt + 通用兜底 prompt

做法：

- 新增独立菜单和专用工作台
- 语种动态读取 `media_languages.enabled=1`
- 常用语种使用内置精调 prompt
- 新增但尚未精调的语种，自动走通用兜底 prompt

优点：

- 最符合当前业务边界
- 页面和逻辑干净
- 动态语种和后续扩展都更稳

缺点：

- 比复用页面多一点实现量

### 结论

采用方案 C。

## 5. 页面与交互设计

### 5.1 菜单与路由

新增独立菜单项，建议名称为：`多语言标题翻译`

路由建议：

- 页面：`GET /title-translate`
- 语言列表接口：`GET /api/title-translate/languages`
- 翻译接口：`POST /api/title-translate/translate`

菜单位置建议放在现有“文案创作 / 文案翻译”附近，作为独立工作台入口。

### 5.2 页面结构

页面采用专用三栏式工作台：

- 顶部：目标语种胶囊区
- 左侧：英文原始数据输入区
- 中间：主操作区，只有一个翻译按钮
- 右侧：翻译结果展示区

桌面端是左右布局，移动端按上下堆叠：

- `< 1024px` 时，三栏改成上下布局
- `< 768px` 时，结果区排到最下方，保证输入和按钮优先可见

### 5.3 语种胶囊

语种来源：

- 查询 `media_languages`
- 过滤条件：`enabled=1`
- 排序：`sort_order ASC, code ASC`
- 页面层排除 `en`

交互规则：

- 单选，不支持多选
- 点击胶囊只切换当前目标语种，不自动触发翻译
- 页面初次加载时默认选中第一个可用语种
- 如果当前没有任何可用目标语种，则进入空态并禁用翻译按钮

### 5.4 输入区

输入区只有一个大文本框，固定接收以下结构：

```text
标题: Bye Bye Wobbly Doors!
文案: That annoying jiggle every time you open a cabinet? GONE. 💨 Fix it PERMANENTLY. Stainless steel strong.
描述: Shop the Fix!
```

规则：

- 字段必须完整：`标题`、`文案`、`描述`
- 顺序必须固定：标题 -> 文案 -> 描述
- 标签必须固定，不接受别名
- 每个字段占一行，不支持字段值跨多行
- 每一项冒号后必须有非空内容
- 不允许出现额外的未标记行
- 不符合规则时，直接报错

页面不做自动纠错，不帮用户重排，也不把自由文本猜测成结构化数据。

### 5.5 翻译按钮

按钮文案随目标语种变化，例如：

- `翻译为德语`
- `翻译为法语`

点击后：

- 前端先做格式校验
- 校验通过才发请求
- 请求中按钮禁用
- 文案切换为 `翻译中...`
- 右侧结果区显示 loading 状态

### 5.6 结果区

结果区固定拆成三块展示：

- 标题
- 文案
- 描述

结果区顶部显示：

- 当前目标语种名称
- 模型名称，例如 `Claude Sonnet`

附加交互：

- 提供一个“复制全部”按钮
- 复制内容为三行结构化结果：

```text
标题: ...
文案: ...
描述: ...
```

### 5.7 三态

页面必须具备完整三态：

- `empty`
  - 右侧提示“翻译结果将显示在这里”
- `loading`
  - 显示轻量 spinner 或骨架
- `error`
  - 显示明确错误信息，不只写“失败”

## 6. 视觉设计约束

遵循项目已有 Ocean Blue Admin 规范：

- 不引入紫色、靛蓝或粉紫渐变
- 卡片白底、浅边框、大圆角
- 胶囊按钮使用海洋蓝 accent
- hover/focus 只动颜色和边框
- 所有颜色和尺寸尽量复用现有 token 或与现有页面风格保持一致

页面风格重点是“清晰、规整、工具感强”，不是营销站视觉。

## 7. 后端设计

### 7.1 页面层

新增一个独立 blueprint，例如：

- `web/routes/title_translate.py`

职责：

- 提供页面模板渲染
- 提供语言列表接口
- 提供翻译接口

第一版不使用 `projects` 表存储历史记录，也不引入后台任务状态。

### 7.2 语言列表接口

`GET /api/title-translate/languages`

返回：

```json
{
  "languages": [
    {"code": "de", "name_zh": "德语", "sort_order": 2},
    {"code": "fr", "name_zh": "法语", "sort_order": 3}
  ]
}
```

规则：

- 使用现有 `appcore.medias.list_languages()` 或等价函数
- 只返回 `enabled=1`
- 过滤掉 `en`

### 7.3 翻译接口

`POST /api/title-translate/translate`

请求体：

```json
{
  "language": "de",
  "source_text": "标题: ...\n文案: ...\n描述: ..."
}
```

返回体：

```json
{
  "result": {
    "title": "...",
    "body": "...",
    "description": "..."
  },
  "language": {
    "code": "de",
    "name_zh": "德语"
  },
  "model": "anthropic/claude-sonnet-4-5"
}
```

### 7.4 模型调用

第一版固定使用 OpenRouter 下的 Sonnet：

- 不给前端 provider 选择
- 不复用 `translate_pref`
- 后端统一走 `openrouter`
- 客户端与 base URL 可复用 `resolve_provider_config("openrouter", user_id=current_user.id)` 的现有能力
- 但最终使用的模型必须强制落在 Claude Sonnet 家族；如果用户把 OpenRouter 自定义模型改成非 Sonnet，则本功能回退到系统默认 `CLAUDE_MODEL`

为了满足“固定 Sonnet”的业务要求，模型来源应限定为 OpenRouter 的 Claude Sonnet 配置。界面不暴露模型切换入口。

### 7.5 prompt 模块

新增专门的 prompt 模块，例如：

- `appcore/title_translate_settings.py`

职责：

- 读取当前可用目标语种
- 返回某个语种对应的默认 prompt

设计原则：

- prompt 不存数据库
- prompt 不开放页面编辑
- prompt 全部由后端内置

### 7.6 prompt 策略

prompt 分为两层：

#### 7.6.1 精调 prompt

对常用语种提供专门模板：

- `de`
- `fr`
- `es`
- `it`
- `ja`
- `pt`

德语 prompt 以用户提供版本为母版。其他语种按同样结构派生，替换：

- 语言角色设定
- 地区化表达习惯
- 当地短视频平台表达节奏
- 合规和消费者偏好描述
- 输出语言标签

#### 7.6.2 通用兜底 prompt

如果后台后续启用了新的目标语种，而代码里没有对应精调模板，则自动生成一份通用 prompt。

通用 prompt 目标：

- 保证功能可用
- 明确要求本土化翻译
- 明确要求输出固定三行结果

这样即使新增语种，也不会导致该页面直接不可用。

### 7.7 prompt 注入方式

默认 prompt 模板里保留一个原文占位符，例如：

- `{{SOURCE_TEXT}}`

后端在校验并解析输入后，将结构化原文重新注入到 prompt 模板中，再发给模型。

这样做的原因：

- 与用户给出的德语 prompt 结构一致
- 语种模板可读性强
- 每个语种都能明确看到“原文区”和“输出格式要求”

### 7.8 模型输出格式

要求模型只返回固定三行：

```text
标题:...
文案:...
描述:...
```

后端收到模型输出后，再做一次解析和严格校验：

- 标签固定使用中文：`标题 / 文案 / 描述`
- 每个字段占一行，不允许跨多行
- 必须能匹配三行
- 字段必须齐全
- 三个值都不能为空

如果模型输出不符合格式，接口返回错误：

- `模型输出格式不合法，请重试`

## 8. 校验规则

### 8.1 输入校验

前端和后端使用相同规则：

- 必须存在 `标题:` 行
- 必须存在 `文案:` 行
- 必须存在 `描述:` 行
- 行顺序固定
- 冒号后内容不能为空
- 不允许缺行
- 不允许换标签名

前端校验失败时不发请求，后端校验失败时返回 400。

### 8.2 语种校验

后端翻译接口收到 `language` 后：

- 必须存在于 `media_languages.enabled=1`
- 必须不等于 `en`

如果页面加载后后台又调整了语种开关，以后端实时校验为准。

### 8.3 输出校验

模型输出必须符合固定结构：

- `标题`
- `文案`
- `描述`

否则按错误处理，不把格式错误结果渲染给用户。

---

## 9. 错误处理

前端错误提示要尽量具体：

- 缺少 `标题:` 行
- 缺少 `文案:` 行
- 缺少 `描述:` 行
- 字段顺序必须为 `标题 -> 文案 -> 描述`
- 每个字段必须填写内容
- 当前目标语种未启用或不存在
- 模型输出格式不合法，请重试
- 调用 Sonnet 失败，请稍后重试

后端不要只返回通用 500 文案，应尽量保留结构化错误消息，便于前端直出。

## 12. 风险与取舍

### 12.1 风险

- 新启用但未精调的语种只能使用通用兜底 prompt，质量可能低于精调语种
- 模型偶发不遵守输出格式时，需要依赖后端严格兜底并提示重试
- 如果 OpenRouter 的用户配置被改成非 Sonnet 模型，会偏离本功能的设计预期

### 12.2 取舍

本次优先保证：

- 入口清晰
- 交互简单
- 语种动态
- 输出稳定

因此主动放弃：

- 历史记录
- prompt 在线编辑
- 多语种并行翻译
- 通用翻译能力复用

---

## 13. 预期结果

完成后，系统会新增一个清晰、专用的“小语种标题文案翻译”工作台：

- 用户从独立菜单进入
- 目标语种自动跟随后台启用配置
- 输入必须是固定三段结构
- 点击按钮后调用 Sonnet
- 结果稳定回到三段结构化输出
- 后续后台新增小语种时，该页面无需改前端结构即可自动显示
