# AutoVideoSrt 本地生产迁移设计

> 目标日期：2026-04-22
> 适用范围：`AutoVideoSrtLocal` 整仓迁移到本地生产服务器 `172.30.254.14`

## 1. 背景

当前仓库已经完成本地数据库导入，`localserver.md` 中保存了本地生产服务器与数据库信息，远程服务器仍保留完整现网环境。接下来的任务不是继续依赖远程环境运行，而是把整套 `AutoVideoSrtLocal` 真正迁到本地服务器，使本地服务器成为唯一正式生产承载机。

这次迁移的范围不是单一模块，而是整仓级别迁移，涵盖：

- Web 应用部署
- 本地 MySQL 接入
- 业务文件主存储切换
- 用户上传入口切换
- 依赖公网回拉的第三方链路保留
- 历史活任务和长期业务数据迁移
- 全模块验收

## 2. 目标

本次迁移完成后，系统应满足以下目标：

- `172.30.254.14` 成为唯一正式生产承载机
- 本地环境运行时只连接本地 MySQL，不再依赖远程数据库
- 业务主文件存储切换到本地文件系统
- 用户上传入口切换为本地服务器直传 `multipart`
- 只有必须公网可回拉的链路继续使用 `TOS` 签名 URL
- 远程服务器保持不动，但不参与正式流量设计
- 整仓所有现有模块完成一轮迁移后验收

## 3. 已确认约束

### 3.1 环境与入口

- 目标服务器：`172.30.254.14`
- 服务器形态：单机承载 `Web 应用 + MySQL + 本地文件存储`
- 操作系统与托管方式：`Linux + systemd`
- 本地 MySQL 服务名：`mysql`
- 生产访问入口：`172.30.254.14` 直连
- 访问性质：`内网 / VPN / 局域网直连`

### 3.2 远程环境策略

- 远程服务器现网先保持不动
- 远程服务器不纳入正式流量设计
- 远程服务器只保留参考价值与应急资料来源
- 本次迁移方案不设计长期双活
- 正式切换后默认禁止两边同时承接正式写入

### 3.3 数据库策略

- 本地数据库已经完成远程数据完整 dump 导入
- 本地环境数据库必须完全切到本地 MySQL
- 运行时不再依赖远程数据库
- 正式切流前只做本地 MySQL 最终复核，不再把远程数据库作为运行依赖或回退依赖

### 3.4 存储策略

- 业务主存储切到本地文件系统
- 本地目录规划尽量沿用现有目录语义，如 `uploads/`、`output/`
- 用户上传入口首版改成本地服务器 `multipart` 直传
- 首版不做分片上传和断点续传
- `TOS` 不再承担业务主存储，只保留“公网交换层”角色
- 必须公网可回拉的链路继续使用 `TOS` 签名 URL

### 3.5 历史数据与文件迁移范围

- 文件迁移来源顺序：先查 `TOS`，缺失再查远程服务器本地磁盘
- 文件迁移执行方：由 `172.30.254.14` 直接拉取
- 只迁“数据库记录中有引用”的文件
- 不迁 `TOS` 桶内孤儿文件
- 历史兼容只保 `活任务`
- `活任务` 定义：`未删除 + 未过期 + created_at 最近 30 天`
- 非任务类长期数据全量迁移
- `TOS` 历史业务文件迁完后先保留观察，不立即删除

### 3.6 验收策略

- 稳定标准：整仓全模块验收
- 验收允许分层：
  - 必须真跑通的真实流程
  - 受外部服务、第三方平台或真人素材约束的冒烟流程

## 4. 当前系统形态

从代码实现看，当前系统处于“本地文件 + TOS 主存储/交换混合态”：

- 新建翻译类任务大量使用 `TOS` 直传入口
- 下载、预览、恢复、清理逻辑普遍把 `TOS` 当成主恢复来源或首选来源
- 部分模块已经把存储语义写进 `projects.state_json`
- 运行时广泛依赖以下状态键：
  - `source_tos_key`
  - `result_tos_key`
  - `tos_uploads`
  - `delivery_mode`
  - `source_object_info`
  - `result_object_info`
- `thumbnail_path` 是列字段且被多处页面直接依赖，必须继续保留

因此，迁移的本质不是简单复制文件，而是要把“主存储判定”和“状态协议语义”从 `TOS` 主导改成本地主导，同时尽量保持既有状态结构和页面协议稳定。

## 5. 目标架构

迁移后的目标架构如下：

- 本地服务器 `172.30.254.14`
  - `systemd` 托管 Web 服务
  - 本地 MySQL 作为唯一正式数据库
  - 本地文件系统作为唯一业务主文件存储
- `TOS`
  - 只用于必须公网回拉的交换型文件
  - 不再作为业务主存储
- 远程服务器
  - 保留现状
  - 不参与正式流量
  - 只在需要时提供历史磁盘补数来源与参考配置

## 6. 存储边界设计

### 6.1 本地文件系统承担的内容

以下内容一律以本地文件系统为准：

- 翻译主链路源视频
- 任务中间产物
- 最终输出视频
- `SRT`
- `CapCut` 导出文件
- 缩略图
- 媒体库长期素材
- 图像翻译结果
- 语音匹配样本与中间文件
- 列表页、详情页和下载接口所需的本地文件

### 6.2 `TOS` 继续承担的内容

以下场景继续保留 `TOS` 签名 URL：

- 豆包 `ASR` 需要第三方主动回拉音频
- 字幕移除类外部服务需要主动回拉源视频
- 视频生成等依赖公网 URL 的外部服务
- 其他明确只接受公网 URL 的第三方链路

### 6.3 首版存储兼容原则

首版不做大规模数据库和状态模型重构，遵循以下原则：

- 现有 `projects` 表结构尽量不动
- `state_json` 继续保留现有关键字段形状
- 旧字段如 `source_tos_key`、`result_tos_key`、`tos_uploads` 可保留为历史映射信息
- 新运行逻辑以本地路径和本地文件存在性为主判断依据
- `thumbnail_path` 继续保持列级兼容

### 8.2 允许按冒烟执行的验收项

冒烟不代表跳过模块，而是允许用“任务可创建、参数可提交、外部链路可到达、错误处理合理”作为首版通过标准。

## 9. 切换流程边界

本次迁移采用“大切换方案”，但不是直接切流，而是按以下边界控制：

- 先在 `172.30.254.14` 完成整仓部署
- 再完成代码适配
- 再完成文件迁移
- 再完成全模块验收
- 最后才把正式生产入口切到本地

切换后：

- 正式生产入口指向本地服务器
- 本地服务器是唯一正式生产运行环境
- 远程服务器不参与正式流量

## 10. 风险

当前已识别的高风险点如下：

- `state_json` 是整份 JSON 读写，多处逻辑共享字段协议，改名和搬字段风险高
- 下载、恢复、清理逻辑普遍默认 `TOS` 可兜底，改成本地优先后需要同步改判定条件
- 部分部署脚本仍硬编码旧服务器地址
- 当前 `gunicorn eventlet` 与 `SocketIO(async_mode="threading")` 并发模型不一致，迁移时必须明确
- 启动阶段会拉起 scheduler 和恢复逻辑，单进程假设必须保持稳定
- 本地环境必须补齐运行时必需环境变量，否则应用无法启动

## 11. 非目标

以下内容明确不属于本次首版目标：

- 长期双活
- 大规模数据库 schema 重构
- 统一对象存储抽象重构
- 分片上传与断点续传
- 立即删除 `TOS` 历史业务文件
- 对所有历史任务做无限期强兼容

## 12. 推荐实施原则

为了让首版更稳，实施时应坚持以下原则：

- 数据库优先稳定，不以 schema 大改为代价换取“语义更干净”
- 运行时优先切主读取来源，不先做大范围重命名
- 用户上传、主链路下载、恢复、清理优先切本地
- 只保留必须的公网交换链路
- 文件迁移优先保障 `活任务` 和长期业务数据
- 验收必须覆盖整仓模块，但允许按外部依赖差异分层执行
