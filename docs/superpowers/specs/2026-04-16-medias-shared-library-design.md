# 素材管理共享库设计

> 日期：2026-04-16
> 状态：已与用户确认，待用户审阅书面 spec

## 1. 背景与目标

当前“素材管理”模块虽然入口对登录用户可见，但实际数据访问仍按创建人隔离：

- 列表接口默认只查当前用户自己的产品素材
- 详情、编辑、删除、上传等接口会校验 `product.user_id == current_user.id`
- 管理员最多只能通过“查看全部”看到全量数据，且部分写操作仍受限

这和业务目标不一致。用户现在要的不是“每人一份素材库”，而是“所有能进入素材管理模块的人看到并操作的是同一份共享素材库”。

本次设计的目标很明确：

- 素材管理改成共享库
- 普通用户和管理员都可以读取、编辑、删除、上传同一份素材数据
- 不再做创建人强制管理
- 先不引入新的模块权限系统，后续如果需要再补

## 2. 范围与非目标

### 2.1 本期范围

- 列表默认返回全部未删除产品，而不是当前用户自己的产品
- 产品详情、编辑、删除、主图上传、视频上传、视频删除全部改为共享可写
- 普通用户与管理员在素材管理中的读写能力一致
- 移除页面上仅管理员可见的“查看全部”开关
- 保留 `user_id` 字段，仅作为审计信息，不再作为权限条件
- 补充测试覆盖共享读写行为

### 2.2 非目标

- 不新增模块级权限表
- 不做“谁能进入素材管理”的白名单控制
- 不做“只读用户”“审核流”“锁定素材”等管理能力
- 不改变其他模块的权限模型

## 3. 方案选型

### 3.1 备选方案

1. 共享读写库：所有登录且可访问 `/medias` 的用户都能查看并修改全部素材
2. 共享只读库：所有人可看，但写操作仍限制创建人或管理员
3. 补一套模块 ACL：显式配置哪些人可以看、哪些人可以改

### 3.2 选择结果

采用方案 1。

原因：

- 与用户要求完全一致：普通用户也能读写，不做强制管理
- 现有系统没有模块 ACL，硬补一套会显著扩大范围
- 共享库本身是当前更符合业务的模型，先把能力做对，比引入额外管理规则更重要

## 4. 权限语义

### 4.1 本期“有权限”的定义

本项目当前没有独立的模块权限系统，因此本期将“有权限”定义为：

- 账户已登录
- 账户处于启用状态
- 能正常访问 `/medias`

这意味着：

- 管理员可以读写共享素材库
- 普通用户也可以读写共享素材库
- 两者在素材管理模块中的权限没有差异

### 4.2 本期不做的限制

以下限制本期全部取消：

- “只能看自己创建的产品”
- “只能修改自己上传的素材”
- “只有管理员才能看全部”
- “管理员只读、不能代替他人修改”

### 4.3 审计字段保留

虽然权限不再按创建人限制，但以下字段继续保留：

- `media_products.user_id`
- `media_items.user_id`

它们仍记录“谁创建了产品 / 谁上传了视频”，用于排查和后续扩展，但不再参与访问控制。

## 5. 后端设计

### 5.1 列表接口

当前：

- `GET /medias/api/products`
- 默认使用 `current_user.id` 调用 `medias.list_products(user_id, ...)`
- 管理员勾选 `scope=all` 才能查看全量

调整后：

- 默认直接返回全部产品
- 不再使用 `current_user.id` 作为列表过滤条件
- 不再保留 `scope=all` 的特殊语义

对应影响：

- `web/routes/medias.py::api_list_products`
- `web/static/medias.js::loadList`
- `web/templates/medias_list.html` 中管理员专属开关

### 5.2 产品访问控制

当前 `_can_access_product(product, write=False)` 的语义是：

- 创建人可读写
- 管理员只在只读场景可跨用户访问

调整后：

- 只要产品存在且未删除，就允许已登录用户访问
- `write=True` 与 `write=False` 不再区分创建人

建议语义：

```python
def _can_access_product(product: dict | None) -> bool:
    return bool(product)
```

这样以下接口都自动共享可写：

- 获取产品详情
- 更新产品
- 删除产品
- 上传 / 替换产品主图
- 上传 / 删除视频素材
- 设置 / 删除视频封面
- 获取视频播放地址、缩略图、主图

### 5.3 DAO 层

`appcore.medias.list_products(user_id, ...)` 保留现有签名即可，但在路由层统一传 `None`。

原因：

- 这样改动更小
- 其他现有测试和调用关系最稳定
- 若以后真的要恢复用户隔离或补 ACL，DAO 不用重新设计

### 5.4 审计与对象路径

上传时对象路径仍可继续使用 `current_user.id` 参与 TOS key 生成，例如：

- `build_media_object_key(current_user.id, pid, filename)`

这是可以接受的，因为：

- 它只是存储路径组成，不影响访问控制
- 不会阻止其他用户读取同一产品里的素材
- 以后若要统一路径结构，可以单独再做迁移

## 6. 前端设计

### 6.1 页面入口

素材管理页面继续保留在左侧导航，仍对登录用户可见，不新增任何权限提示。

### 6.2 顶部筛选区

当前页面上存在管理员专属的“查看全部”开关。

调整后：

- 该开关移除
- 所有人默认就是“查看全部”

保留：

- 关键字搜索
- 已归档筛选

### 6.3 交互语义

前端无需再区分“我的素材”和“全部素材”，也不需要判断管理员特殊行为。

前端统一按共享库处理：

- 列表展示所有产品
- 任意产品都可点开编辑
- 任意产品都可修改文案、主图、视频素材
- 任意产品都可删除

## 7. 数据与兼容性

### 7.1 既有数据

现有 `media_products` 和 `media_items` 表中的历史数据无需迁移。

改动生效后：

- 之前由 A 用户创建的产品，将直接对 B、C、D 用户可见
- 之前由 A 用户上传的视频，也将成为共享素材的一部分

### 7.2 与其他模块的边界

本次只改变素材管理模块的共享语义，不影响：

- 视频翻译任务
- 文案创作任务
- 字幕移除任务
- API 配置
- 用量统计

这些模块仍按各自当前的用户隔离逻辑执行。

## 8. 测试策略

本次至少补以下测试：

### 8.1 路由层

- 普通用户访问 `/medias/api/products` 时能拿到他人创建的产品
- 普通用户访问 `/medias/api/products/<pid>` 时能读取他人产品详情
- 普通用户可以更新他人产品
- 普通用户可以删除他人产品

### 8.2 页面结构

- 素材管理页不再渲染“查看全部”开关
- 前端不再依赖 `MEDIAS_IS_ADMIN` 来决定是否传 `scope=all`

### 8.3 回归

- 创建产品仍正常
- 产品 ID 唯一性校验仍正常
- 编辑页和新增页已有布局测试继续通过

## 9. 风险与取舍

### 9.1 风险

共享可写意味着任意登录用户都能修改或删除共享素材，这会降低权限隔离。

### 9.2 本期取舍

这是一个明确接受的业务取舍，而不是漏洞：

- 用户已经确认“普通用户也能读写，不做强制管理”
- 当前优先级是让素材管理变成共享工作台
- 若后续需要更细的权限，再单独设计 ACL 或审核流

### 9.3 回退成本

本次改动主要集中在路由和页面条件渲染层，若未来要恢复更细粒度权限：

- 可以重新启用 `_can_access_product` 的差异逻辑
- 可以重新为列表接口加入范围过滤
- 可以补一张模块权限表，并在路由层替换当前“登录即可”的判断

因此本次方案是可回退、可演进的。
