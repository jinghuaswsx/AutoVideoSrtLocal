# 链接检查项目化与失败高亮设计

## 背景

当前“链接检查”模块还是一次性页面任务：

- 任务状态主要保存在运行时内存结构中
- 页面刷新后只能依赖当前任务接口继续看结果
- 任务列表、项目详情、后续跟进能力缺失
- 结果页虽然有完整字段，但未通过项不够醒目，排查效率低
- 目标语言需要人工选择，无法根据 Shopify 多语言链接自动带出

这与团队当前的实际使用方式不匹配。链接检查已经不是临时工具，而是一个需要持续跟进、反复复查、多人协作查看的模块。

## 目标

本次设计要同时解决两类问题：

1. 将“链接检查”升级为真正的项目模块
2. 让未通过原因在详情页里一眼可见

明确目标如下：

- `链接检查` 改为全局可见的项目列表 + 项目详情模式
- 不做用户隔离；所有登录用户都能看到并使用这些项目
- 检查过程中的中间状态全部持久化，方便后续继续跟进
- 详情页中未通过图片要整卡高亮，失败字段和依据要标红、加粗、放大
- 新建项目时，目标语言应根据输入链接自动识别并带出
- 自动识别语言的映射来源仍然是 `media_languages`

## 非目标

本次不包含以下能力：

- 复杂权限模型
- 项目标签、备注、负责人流转
- 批量删除、批量重跑
- 结果页自定义筛选器
- 单独的新数据库表体系

## 推荐方案

推荐直接沿用现有项目体系，而不是给链接检查单独重做一套存储。

核心思路：

- 使用现有 `projects` 表承载 `link_check` 项目
- `state_json` 保存链接检查全过程状态
- `/link-check` 从单页检查页改为“项目列表页”
- 新建项目后跳转到 `/link-check/<task_id>` 详情页
- 详情页持续轮询或刷新项目状态，展示抓图、识别、匹配、汇总结果

这样能最大化复用现有“视频翻译 / 视频创作 / 视频评估”的项目模式，同时把“中间状态保留”和“后续跟进”自然接入。

## 项目模型

### 项目类型

在 `projects` 表中新增或复用：

- `type = 'link_check'`

该类型项目不再走当前的临时内存任务入口，而是走数据库持久化项目流。

### 可见性

链接检查项目是全局共享的：

- 列表页不按 `user_id` 过滤
- 详情页不按 `user_id` 校验拥有关系
- 所有登录用户都可以查看、创建、删除、重试

`user_id` 仍保留创建人信息，作为审计字段，而不是权限字段。

### 保留策略

链接检查项目默认永久保留，不走常规自动过期清理。

具体口径：

- `expires_at = NULL`
- 清理任务默认跳过 `type='link_check'`
- 项目删除改为显式人工删除

这样才能满足“中间状态全部保留，方便后续跟进”的要求。

## 状态持久化

`state_json` 复用当前运行时结构，但改为数据库持续写回。

建议状态结构保留这些一级字段：

- `link_url`
- `resolved_url`
- `page_language`
- `target_language`
- `target_language_name`
- `progress`
- `summary`
- `reference_images`
- `items`
- `error`
- `steps`

其中：

- `progress` 记录抓图总数、已分析数、已比较数、二值快检数、同图大模型数
- `items` 保存每张图片的完整检查结果
- `steps` 记录项目级阶段，例如：
  - `create`
  - `lock_locale`
  - `download`
  - `analyze`
  - `finalize`

每完成一个阶段、每分析完一张图，就立即写回 `projects.state_json`，避免刷新丢上下文。

## 页面结构

### 列表页

`/link-check`

改为项目列表页，风格跟现有项目页保持一致：

- 顶部标题：`链接检查`
- 右上角按钮：`+ 新建项目`
- 卡片/列表双视图可复用现有模式
- 每个项目展示：
  - 项目名
  - 原始链接
  - 目标语言
  - 当前状态
  - 创建时间
  - 整体结论
  - 未通过图片数

空状态文案改为“还没有链接检查项目，点击右上角新建”。

### 新建项目

新建项目用弹窗或页内面板，输入项为：

- 检查链接
- 目标语言
- 参考图片（可选）

目标语言支持自动带出，但仍允许人工改。

### 详情页

`/link-check/<task_id>`

详情页承担原来单页的全部功能，并增加项目上下文：

- 顶部显示项目基础信息与整体结论
- 中间显示进度与汇总统计
- 下方按图片卡片展示每张结果
- 保留现有“查看任务详情”弹窗

## 失败高亮规则

### 卡片级高亮

对于以下图片卡片，整卡高亮：

- `analysis.decision in {'replace', 'review'}`
- `item.status == 'failed'`

视觉规则：

- 浅红背景
- 明显红色边框
- 顶部增加 `问题摘要` 区

### 字段级高亮

以下字段命中失败条件时，值和依据同时高亮：

- `最终判定`
- `识别语言`
- `质量分`
- `二值快检结果`
- `二值相似度`
- `前景重合度`
- `大模型相同图片判断`
- `模型说明`
- `二值快检说明`

视觉规则：

- 红色文字
- 更大字号
- 加粗

### 问题摘要

每张未通过卡片顶部增加一条摘要，聚合失败原因，例如：

- `语言不匹配`
- `二值快检未通过`
- `前景重合度低于阈值`
- `质量分过低`
- `模型判定需替换`

摘要来源于当前已有字段推导，不引入新的模型调用。

## 目标语言自动识别

### 识别来源

当用户输入链接时，从 URL path 中自动提取语言片段。

优先支持：

- `/fr/`
- `/de/`
- `/ja/`

同时兼容 Shopify Markets 常见格式：

- `/fr-fr/`
- `/de-de/`
- `/en-de/`

### 识别规则

1. 解析 URL path 的第一层 locale-like segment
2. 如果该 segment 在 `media_languages.code` 中有精确匹配，直接使用
3. 如果是 `xx-yy` 结构，则优先取第一个子标签 `xx` 再去匹配 `media_languages`
4. 如果没有命中启用语种，不自动选中，保留人工选择

示例：

- `/fr/products/...` -> `fr`
- `/fr-fr/products/...` -> `fr`
- `/en-de/products/...` -> `en`

这里的判断口径是“页面语言”，不是市场国家。

### 前端交互

- 用户输入或粘贴链接后，自动尝试识别语言
- 如果识别成功，自动选中目标语言下拉
- 如果用户手动改过目标语言，则后续不强制覆盖，只提示“已从链接识别到语言”

## 后端结构调整

### 路由

现有 [web/routes/link_check.py](/g:/Code/AutoVideoSrt/web/routes/link_check.py) 需要拆成两类职责：

- 项目页面路由
  - 列表页
  - 详情页
- 项目 API
  - 新建项目
  - 读取项目状态
  - 删除项目
  - 重试项目
  - 读取站点图 / 参考图

### 状态来源

当前 `store.create_link_check()` 和 `task_state.create_link_check()` 的内存任务流，需要迁移到数据库项目流。

建议新增一个持久化服务层，负责：

- 创建 link-check 项目
- 更新 `state_json`
- 查询全局项目列表
- 查询详情

运行时 [appcore/link_check_runtime.py](/g:/Code/AutoVideoSrt/appcore/link_check_runtime.py) 不再只操作内存任务，而是改为对数据库项目持续写回。

## 测试要求

至少补齐以下测试：

- 列表页只展示 `type='link_check'` 项目
- 列表页不按 `user_id` 过滤
- 新建项目后写入 `projects`
- `state_json` 在中间阶段持续更新
- `expires_at` 对 `link_check` 为空
- 目标语言能从 `/fr/`、`/fr-fr/` 这类链接自动识别
- 失败卡片字段高亮的前端资产测试
- 未通过图片数和整体结论的列表展示测试

## 风险与取舍

### 风险 1：项目永久保留导致存储增长

这是有意选择，因为业务明确要求保留中间状态方便跟进。

当前先接受这个取舍，后续如果量级上来，再补后台人工清理或归档策略。

### 风险 2：全局共享缺乏权限隔离

这是当前业务明确要求，不做用户隔离。

后续如果出现误删或审计需求，再补创建人、操作日志和只读权限。

### 风险 3：语言目录格式不统一

Shopify 的语言/市场 URL 既可能是单语言，也可能是语言+国家组合目录，因此自动识别必须允许失败并保留人工覆盖，不能写死成只认 `/fr/` 一种格式。

## 结论

本次推荐将“链接检查”升级为复用现有 `projects` 表的全局项目模块，并同时补齐：

- 全局项目列表与详情
- 中间状态持久化
- 失败卡片与失败字段高亮
- 链接语言自动识别

这是当前成本最低、与现有架构最兼容、也最符合团队后续跟进方式的方案。
