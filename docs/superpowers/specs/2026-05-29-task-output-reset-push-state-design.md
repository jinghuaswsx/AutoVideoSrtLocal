## 背景

推送管理的素材状态由 `media_items.pushed_at` 和 `latest_push_id` 计算。任务中心重做小语种视频产物时，如果复用同产品同语种的已有 `media_items` 行并覆盖文件信息，旧行上的 `pushed_at` 会继续让推送管理显示“已推送”，导致新产物无法自然进入“待推送”队列。

本次反馈中的 `bell-rjc` 德语、法语、西班牙语视频已经重新做好，但推送管理仍显示已推送，符合上述状态未失效场景。

## 目标

1. 任务中心重新提交翻译视频后，新产物不能继承旧素材的已推送状态。
2. 历史推送日志和旧素材的 `pushed_at/latest_push_id` 保留，用于审计旧版本推送记录。
3. 新视频产物由现有推送状态计算决定展示为“待推送”或“未就绪”。
4. 不在任务刚创建或重发时清状态，避免新产物尚未提交时把旧素材提前放回待推送。

## 行为规则

- 任务中心手工提交新的视频产物时采用 append-only：始终创建新的 `media_items` 行，并通过 `task_id` 绑定当前任务产物。
- 若同产品同语种存在旧 `media_items`，即使旧行已推送，也不覆盖旧行、不复用旧行、不继承旧行的 `pushed_at/latest_push_id`。
- 新建 `media_items` 行默认没有推送锁，因此会按 readiness 自然显示为“待推送”或“未就绪”。
- 不清空 `skip_push`，因为“不推送”是管理员的显式决定，不应被重传动作隐式覆盖。
- 推送历史 `media_push_logs` 不删除，仍可用于审计旧版本推送记录。

## 验证

1. 单测覆盖：同产品同语种已有已推送视频 item 时，任务中心重传视频会追加新的 `media_items` 行，不覆盖旧行。
2. 聚焦运行相关 `pytest`。
3. 编译检查变更的 Python 文件。
