# AutoPush 小语种文案独立推送设计

**日期**：2026-04-21
**状态**：已确认，待实现
**适用范围**：`AutoPush/` 子项目 + 主项目 OpenAPI 推送上下文接口

## 背景

当前 AutoPush 弹窗只有两块能力：

- `推送确认`
- `JSON 预览`

二者都服务于同一条素材推送链路：AutoPush 从主项目 OpenAPI 拉取素材推送 payload，然后把该 payload 直发到素材推送接口。

现在需要在点击推送后的弹窗顶部增加第三个胶囊按钮 `推送小语种文案`。该按钮不是单纯的查看视图，而是对应一条**独立的推送链路**：

- 素材推送：继续走现有素材推送接口
- 小语种文案推送：单独调用 `POST /api/marketing/medias/{mk_id}/texts`

同时，小语种文案推送报文中的 `texts[0]` 不能再使用占位值，而必须来自“这条视频素材当前语种对应的文案第一条”。

## 目标

1. AutoPush 弹窗头部改成 3 个胶囊按钮：
   - `推送确认`
   - `JSON 预览`
   - `推送小语种文案`
2. 同一个弹窗底部仍保留一个 `推送` 按钮，但点击行为随当前胶囊切换：
   - `推送确认` 和 `JSON 预览`：调用现有素材推送接口
   - `推送小语种文案`：调用小语种文案接口
3. 小语种文案推送报文由主项目提供的数据上下文组装，文案内容取当前素材语种的首条文案。
4. 如果缺少 `mk_id`，则直接报错并拒绝推送，不提供手动填写入口。

## 非目标

- 不在弹窗内编辑小语种文案
- 不允许用户手工录入 `mk_id`
- 不把 `JSON 预览` 改造成同时预览两种不同接口的请求体
- 不改变现有素材推送接口的下游契约

## 用户交互

### 弹窗顶部

弹窗头部改为三个并列胶囊，表现为互斥切换：

- `推送确认`
- `JSON 预览`
- `推送小语种文案`

默认打开时选中 `推送确认`。

### 各胶囊对应内容

#### 1. 推送确认

展示当前素材的推送确认信息，沿用现有结构化内容：

- 产品
- 语种
- 文件
- item_id
- 状态
- 现有素材推送 payload 的结构化摘要

点击底部 `推送` 时，走现有素材推送链路。

#### 2. JSON 预览

只展示**素材推送**的完整 JSON payload。

说明：

- 不展示小语种文案接口的请求体
- 该胶囊与 `推送确认` 对应的是同一条素材推送链路，只是展示方式不同

点击底部 `推送` 时，仍走现有素材推送链路。

#### 3. 推送小语种文案

展示当前素材当前语种的第一条文案，字段为：

- `语种`
- `标题`
- `文案`
- `描述`

点击底部 `推送` 时，走独立的小语种文案接口。

## 数据来源

### 小语种文案内容

数据来源是：

- 当前素材所属产品
- 当前素材语种 `item.lang`
- `media_copywritings` 中该语种的第一条文案

映射规则：

- `title` <- 文案标题
- `message` <- 文案正文 `body`
- `description` <- 文案描述
- `lang` <- 当前语种的人类可读名称

`lang` 的显示和值，优先使用系统语言表里的中文名；拿不到时退回语言 code。

### mk_id

`mk_id` 由后续适配接口返回，字段名固定为 `mk_id`。

设计假设：

- 联调时主项目 OpenAPI 返回的推送上下文里会包含 `mk_id`
- AutoPush 不负责自行查询或推导 `mk_id`

## 两条推送链路

### 链路 A：素材推送

继续沿用现有流程：

1. AutoPush 从主项目拉取素材推送 payload
2. 点击 `推送` 后，POST 到现有素材推送目标
3. 成功后继续走现有写回逻辑

该链路下的 `JSON 预览` 仍然对应素材推送 payload。

### 链路 B：小语种文案推送

当当前胶囊为 `推送小语种文案` 时：

1. 读取当前推送上下文中的 `mk_id`
2. 读取当前素材当前语种的首条文案
3. 组装请求体：

```json
{
  "texts": [
    {
      "title": "Ready. Aim. LAUNCH! 🌪️",
      "message": "Experience the thrill! 🤩 Instant mechanical launch. Durable & crash-proof. The coolest gift for ages 3+.",
      "description": "Fly High Today ✈️",
      "lang": "法语"
    }
  ]
}
```

4. 调用：

```text
POST https://os.wedev.vip/api/marketing/medias/{mk_id}/texts
```

5. 在弹窗响应区展示这次文案推送的响应结果

## 错误处理

### 缺少 mk_id

这是硬阻断条件。

表现：

- 切到 `推送小语种文案` 时，如果没有 `mk_id`，显示明确错误
- 点击 `推送` 时直接拒绝，不做任何降级或兜底

建议文案：

```text
缺少 mk_id，暂不能推送小语种文案
```

### 缺少当前语种首条文案

如果当前语种没有可用的第一条文案：

- `推送小语种文案` 视图展示空状态
- 点击 `推送` 时拒绝执行

### 两条推送链路必须分开记账

- 素材推送成功或失败，不能误显示成文案推送结果
- 文案推送成功或失败，也不能覆盖素材推送结果的展示语义

至少在当前弹窗会话中，需要明确显示本次请求对应的是哪条链路。

## 主项目接口改动

主项目需要提供一份足够给 AutoPush 使用的推送上下文。该上下文除了现有素材推送 payload 以外，还需要包含：

- `mk_id`
- 当前素材当前语种首条文案的结构化数据

推荐形式：

```json
{
  "item_id": 238,
  "mk_id": 3725,
  "payload": {
    "...": "现有素材推送 payload"
  },
  "localized_text": {
    "title": "Ready. Aim. LAUNCH! 🌪️",
    "message": "Experience the thrill! 🤩 Instant mechanical launch. Durable & crash-proof. The coolest gift for ages 3+.",
    "description": "Fly High Today ✈️",
    "lang": "法语"
  }
}
```

如果为了兼容旧接口，也可以继续把文案放进 `payload.texts[0]`，但 AutoPush 在代码层应把“小语种文案推送数据”和“素材推送 payload”视为两个独立概念，避免未来接口继续分叉时耦合。

## AutoPush 前端实现要点

1. 弹窗 header 从“标题 + JSON 按钮”改为三个胶囊切换
2. 维护当前活动胶囊状态
3. 底部 `推送` 按钮根据当前胶囊分派到不同请求函数
4. 文案推送请求函数单独校验：
   - 是否有 `mk_id`
   - 是否有当前语种首条文案
5. 响应区保留，但要清楚说明当前展示的是素材推送响应还是文案推送响应

## 测试计划

### 主项目

1. 能返回当前语种首条文案的结构化数据
2. `mk_id` 缺失时，上下文仍可返回，但前端文案推送必须拒绝
3. 当前语种没有首条文案时，文案推送上下文为缺失状态

### AutoPush

1. 三个胶囊切换正确
2. 在 `推送确认` 下点击 `推送`，调用现有素材推送接口
3. 在 `JSON 预览` 下点击 `推送`，仍调用现有素材推送接口
4. 在 `推送小语种文案` 下点击 `推送`，调用 `/api/marketing/medias/{mk_id}/texts`
5. `mk_id` 缺失时，文案推送报错并拒绝发请求
6. 当前语种首条文案缺失时，文案推送报错并拒绝发请求
7. 素材推送与文案推送的响应展示不串线

## 风险与约束

1. `mk_id` 联调前可能暂时缺失，因此文案推送在缺少 `mk_id` 的环境中会始终不可用
2. 素材推送 payload 与文案推送 payload 已经分叉，后续若接口继续演进，不应再把两者混成一个统一预览块
3. 如果主项目返回的小语种文案只在 `payload.texts[0]` 中存在，而没有独立字段，AutoPush 代码需要做清晰封装，避免把“展示来源”和“素材推送载荷”强耦合在一起

## 验收口径

1. 弹窗头部显示三个胶囊按钮
2. `推送确认` 和 `JSON 预览` 下仍可正常走素材推送
3. `推送小语种文案` 下点击 `推送` 时，发起 `POST /api/marketing/medias/{mk_id}/texts`
4. 小语种文案内容来自当前语种第一条文案
5. 缺少 `mk_id` 时直接报错，拒绝推送
