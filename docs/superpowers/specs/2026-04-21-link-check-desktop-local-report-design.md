# 链接检查桌面端本地结果页设计

日期：2026-04-21

## 背景

当前桌面端已经能完成以下最小闭环：

1. 通过服务端 OpenAPI 识别产品、语言和参考图
2. 用可见浏览器访问目标页面并下载网站图片
3. 执行参考图匹配、二值快检、同图大模型判断和语言分析
4. 将任务结果写入 `img/<product_id>-<timestamp>/`

但用户拿到结果时仍主要依赖 GUI 文本提示和若干 JSON 文件，不利于人工快速复核。现有 Web 版链接检查结果页的信息组织已经比较成熟，因此桌面端需要补一份“本地静态结果页”，让用户在任务结束后直接打开 `report.html` 查看结果。

## 目标

本次新增能力目标如下：

1. 每次桌面端成功执行后，在任务目录内生成 `report.html`
2. `report.html` 不依赖本地 HTTP 服务，直接双击即可打开
3. 页面整体信息组织接近现有链接检查详情页
4. 图片结果区域采用用户指定的“单行一图”布局
5. 任务完成后自动打开本地结果页
6. GUI 中同步显示结果页路径，方便用户二次打开

## 非目标

本次明确不做：

1. 不复用现有 Web 详情页的轮询逻辑
2. 不新增本地 Web 服务
3. 不将本地结果页做成前后端分离的多文件站点
4. 不支持任务运行中的增量刷新
5. 不重新设计一套新的检测字段，只消费现有桌面端结果

## 方案选择

### 方案 1：单文件静态 HTML

在任务结束后，由桌面端把结果数据直接写入一份自包含的 `report.html`。页面内联 CSS 和渲染所需的少量脚本，图片通过相对路径引用任务目录中的 `site/` 和 `reference/` 文件。

优点：

1. 双击即可打开，最符合“本地 html”诉求
2. 与绿色包分发方式天然兼容
3. 拷贝整个任务目录到其他机器后仍可查看

缺点：

1. 需要在桌面端维护一份静态页面生成逻辑

### 方案 2：HTML + JSON + 独立 JS/CSS

生成 `report.html`、`report.json`、`report.js`、`report.css` 多文件，由浏览器通过 `file://` 加载 JSON 渲染。

优点：

1. 页面资源拆分更清晰

缺点：

1. 不同浏览器对 `file://` 读取本地 JSON 的限制不一致
2. 兼容性和排障成本更高

结论：采用方案 1。

## 页面结构

### 顶部摘要区

页面顶部保留简洁摘要，显示：

1. 产品名称
2. 产品 ID
3. 目标语言
4. 最终访问链接
5. 总图片数
6. 通过 / 需替换 / 待复核数量
7. 整体结论

### 图片结果区

每张网站图片渲染为一行结果卡片，结构如下：

1. 左侧固定两个 `200x200` 预览框
2. 第一个预览框显示网站图
3. 第二个预览框显示匹配到的参考图
4. 若没有参考图或未匹配成功，则第二个预览框留空并显示“无参考图”
5. 图片使用固定容器、等比缩放、居中显示，避免撑高布局

### 右侧关键信息区

右侧采用紧凑三层排布：

1. 第一层为 badge 区
   - 图片类型
   - 最终判定
   - 参考图匹配状态
2. 第二层为核心检测值
   - 识别语言
   - 质量分
   - 二值快检结果
   - 二值相似度
   - 前景重合度
   - 同图大模型结果
3. 第三层为一句话说明
   - 质量说明或最终判定原因
   - 最多显示两行

### 折叠详情

每行卡片提供一个折叠详情区，用于承载次要但仍有排查价值的信息：

1. 原始图片 URL
2. 最终下载 URL
3. 是否保持同一资源
4. 是否来自当前 Variant
5. 大模型通道
6. 大模型名称
7. 下载证据说明

## 数据来源

结果页不重新请求服务端，仅消费桌面端当前已经具备的数据：

1. `bootstrap` 中的产品与语言信息
2. `page_result` 中的页面访问结果
3. `analysis` 中的摘要与逐图结果
4. 本地图片路径

为避免浏览器跨目录权限问题，页面中的图片地址统一写成相对 `report.html` 的相对路径。

## 输出约定

任务目录结构新增：

```text
img/<product_id>-<YYYYMMDDHHMMSS>/
  report.html
  task.json
  page.html
  page_info.json
  reference/
  site/
  compare/
    result.json
```

同时在控制器返回结果中新增：

1. `report_html_path`

## 自动打开

桌面端 GUI 在任务成功结束后：

1. 显示本地结果页路径
2. 自动用系统默认浏览器打开 `report.html`

如果浏览器自动打开失败，不影响任务成功，只在 GUI 中保留结果页路径供用户手动打开。

## 失败处理

本次先聚焦成功结果页闭环：

1. 仅在成功完成完整分析后生成 `report.html`
2. 失败时保留现有 GUI 错误提示和 JSON 落盘行为
3. 失败版静态结果页后续如有需要再补
