# 素材管理原始去字幕素材弹窗改造设计

- 日期：2026-04-22
- 状态：Design（待实现）
- 作者：Codex

## 背景 / 问题

当前素材管理页中的“原始去字幕素材”入口使用右侧抽屉展示，信息密度偏低，主要以单行列表呈现：

- 预览区域只显示小缩略图，无法直接复刻编辑页里的 9:16 素材查看体验
- “查看视频 / 查看封面”依赖外链跳转，用户无法在当前上下文里直接预览与播放
- 抽屉宽度固定且偏窄，不适合承载素材卡片式浏览

用户希望把这一块改成更接近编辑页的素材查看方式，核心要求已经确认：

1. “原始去字幕素材”从右侧抽屉改为居中大弹窗
2. 弹窗宽度约占浏览器页面的 2/3
3. 每条素材卡内部提供“封面图 / 视频”两个 tab
4. 预览框使用适合竖版素材的 `180x320`、`9:16` 视觉框
5. 视频 tab 中可直接播放，整体效果复刻编辑页现有素材展现方式

## 目标

1. 把“原始去字幕素材”入口改造成居中 modal，而不是右侧 drawer
2. 复用编辑页现有的视频素材卡结构、交互语义和加载方式
3. 每条原始素材都能在卡片内完成封面预览和视频播放，不再依赖外链查看
4. 保留当前已有上传、删除、列表刷新和计数同步能力
5. 在桌面端和中小屏下都维持清晰的 9:16 预览体验

## 非目标

- 不改动原始去字幕素材的后端数据模型与接口结构
- 不新增“编辑原始素材封面”“重命名原始素材”等新能力
- 不在这次改造里重做上传接口，只调整上传弹窗表现层
- 不把编辑页和原始素材页抽象成新的通用组件体系，先以复用样式和渲染逻辑为主

## 现状与复用基线

### 当前原始去字幕素材入口

素材管理页模板 [web/templates/medias_list.html](../../../web/templates/medias_list.html) 当前包含：

- `#rsDrawerMask` / `#rsDrawer`：右侧抽屉容器
- `#rsList`：原始素材列表
- `#rsUploadMask`：原始素材上传弹窗

前端逻辑在 [web/static/medias.js](../../../web/static/medias.js) 中：

- `openRawSourceDrawer`
- `closeRawSourceDrawer`
- `refreshRawSourceList`
- `renderRawSourceRow`
- `submitRawSourceUpload`
- `deleteRawSource`

其中 `renderRawSourceRow` 目前渲染的是“缩略图 + 标题 + 外链按钮 + 删除按钮”的单行结构。

### 可复用的编辑页素材卡实现

编辑页已经有一套接近目标效果的实现，可直接作为视觉与交互基线：

- 模板和样式入口：
  - [web/templates/medias_list.html](../../../web/templates/medias_list.html)
  - [web/templates/_medias_edit_detail_modal.html](../../../web/templates/_medias_edit_detail_modal.html)
- 相关样式：
  - `.oc-vitem`
  - `.vtabs`
  - `.vtab`
  - `.vbody`
  - `.vpane`
- 相关脚本：
  - `edRenderItems`
  - `edEnsureVideoLoaded`

这套实现已经满足以下目标能力：

- 卡片内双 tab 切换“图片 / 视频”
- 视频首次点击时懒加载播放地址
- 使用 `<video controls preload="metadata">` 直接播放
- 预览区域采用 `aspect-ratio: 9 / 16`

本次原始去字幕素材弹窗应优先复用这套结构，而不是重新发明新的预览模式。

## 方案选型

### 方案 A：抽屉改居中大弹窗，并复用编辑页卡片结构（推荐）

做法：

- 将当前 `rsDrawer` 结构改为标准 modal
- 主体改为卡片网格，而不是单行列表
- 每张卡片使用“封面图 / 视频”双 tab
- 视频 tab 内直接播放

优点：

- 最符合用户要求
- 和编辑页视觉、交互一致
- 现有样式和脚本复用度最高

代价：

- 需要替换原有 drawer 相关 CSS 与 DOM 结构

### 方案 B：保留右侧抽屉，只放大尺寸

做法：

- 保留 drawer 动画和布局
- 仅扩大宽度并在内部塞卡片网格

问题：

- 本质仍是抽屉，不符合“弹窗占页面 2/3”的要求
- 视觉上仍然不像编辑页

### 方案 C：左侧列表 + 右侧详情预览

做法：

- 左侧展示原始素材列表
- 右侧展示当前选中素材的封面 / 视频 tab

问题：

- 交互模式偏离“复刻编辑页”
- 对素材数量不多的场景来说复杂度过高

结论：采用方案 A。

## 交互设计

## 弹窗容器

“原始去字幕素材”入口改为居中 modal，布局原则如下：

- 宽度：`min(66vw, 1200px)`
- 小屏兜底：`min(92vw, 1200px)`
- 高度：限制在视口内，避免溢出屏幕
- 内容区垂直滚动，头部固定在弹窗内部顶部

弹窗结构仍保留三段：

1. 头部：标题、摘要、关闭按钮
2. 工具栏：上传素材按钮
3. 主体：原始素材卡片网格

标题与摘要语义保持不变：

- 标题：`原始去字幕素材`
- 摘要：`{产品名} · 共 N 条原始去字幕素材`

## 原始素材卡片

每条素材渲染为一张卡片，遵循编辑页的视频素材卡结构：

1. 标题区
2. tab 区
3. 9:16 预览区
4. 元信息区
5. 操作区

### 卡片尺寸

- 卡片宽度：`180px`
- 预览框尺寸：`180x320`
- 预览框比例：`9:16`

说明：

- 编辑页当前 `aspect-ratio: 9 / 16` 的表现可以保留
- 但原始素材弹窗这里应把外层卡片宽度收敛到 `180px`，避免一屏只能放两张大卡
- 预览框必须稳定为竖版，不允许因内容拉伸导致比例变形

### Tab 设计

每张卡片内部提供两个 tab：

- `封面图`
- `视频`

默认激活 `封面图`。

切换规则：

- 点击 `封面图`：显示图片 pane
- 点击 `视频`：首次进入时请求播放地址，显示 loading；成功后渲染 `<video controls>`，失败时仅在当前 pane 内显示错误提示

### 预览内容

封面图 pane：

- 优先使用 `cover_url`
- 如果没有封面图，显示占位态

视频 pane：

- 首次切换到视频时调用现有播放地址接口
- 成功后插入 `<video controls preload="metadata" src="...">`
- 用户点击播放器即可播放

### 元信息

卡片标题下方或底部展示轻量元信息：

- 时长：沿用现有 `fmtRawDuration`
- 大小：沿用现有 `fmtRawSize`

不再保留“查看视频 / 查看封面”这两个链接按钮，因为 tab 预览已经覆盖主诉求。

### 操作区

本次卡片底部只保留必要管理动作：

- `删除`

删除行为沿用现有 `deleteRawSource` 逻辑。

## 上传弹窗

上传仍沿用当前独立弹窗入口，不合并到主弹窗主体中，但上传表单的视觉结构需要更接近编辑页新增素材区。

建议结构：

1. 左侧：封面图 9:16 上传预览框
2. 右侧：视频文件选择框
3. 下方：名称输入框
4. 底部：取消 / 提交按钮

约束：

- 仍然要求视频与封面均为必填
- 视频和封面文件类型校验沿用现有接口约束
- 这次改造不新增自动抽帧封面等能力

该部分目标是让上传体验与查看弹窗保持风格一致，而不是引入新的上传流程。

## 状态设计

### 主弹窗

加载态：

- 打开弹窗后，摘要先显示“加载中”
- 主列表区域显示轻量 loading 占位，不要整屏闪烁

空状态：

- 当 `items.length === 0` 时，显示居中的空状态块
- 文案应明确引导用户先上传第一条原始去字幕素材

错误态：

- 若原始素材列表请求失败，在主列表区域显示错误提示和可重试入口
- 错误不应导致整个页面卡死

### 单卡视频 pane

加载态：

- 点击 `视频` tab 后，仅当前卡片的视频 pane 显示“加载中”

错误态：

- 仅当前卡片的视频 pane 显示“视频加载失败”
- 其他卡片不受影响

## 响应式设计

桌面端：

- 弹窗宽度接近页面 2/3
- 卡片以多列网格展示

中等屏幕：

- 弹窗宽度自动收敛到 `92vw` 左右
- 卡片网格自动换列

窄屏：

- 允许退化为单列或双列
- 单卡预览仍保持 9:16，不允许压扁

这次不要求把主弹窗彻底重做成移动端独立全屏页面，但必须保证不会因固定宽度导致内容溢出不可用。

## 数据与接口

本次设计不改接口协议，继续复用以下 API：

- `GET /medias/api/products/{pid}/raw-sources`
- `POST /medias/api/products/{pid}/raw-sources`
- `DELETE /medias/api/raw-sources/{rid}`

视频播放地址请求优先复用编辑页现有的“点视频 tab 后再拉地址”的模式。如果原始去字幕素材与编辑页素材的接口字段不同，则只在前端渲染层做最小适配，不扩展新协议。

## 实现边界

建议改动聚焦在以下文件：

- [web/templates/medias_list.html](../../../web/templates/medias_list.html)
  - 把 `rsDrawer` 相关 DOM 改为大弹窗结构
  - 调整上传弹窗结构
- [web/static/medias.js](../../../web/static/medias.js)
  - 替换 `renderRawSourceRow`
  - 新增原始素材卡 tab 切换逻辑
  - 新增原始素材视频懒加载逻辑
  - 保持上传、删除、刷新、计数同步

必要时可以在 `medias_list.html` 中新增一组独立的 raw-source modal 样式，但应尽量复用现有：

- `.oc-vitem`
- `.vtabs`
- `.vtab`
- `.vbody`
- `.vpane`

如果复用编辑页样式会互相污染，则优先抽出“原始素材卡专用样式”并保持命名清晰，不要直接把编辑页样式改得过于耦合。

## 测试与验证

### 自动化

本次前端改造至少需要补齐与确认：

- 原始素材列表接口相关测试继续通过
- 原始素材弹窗渲染相关前端逻辑不会破坏现有上传 / 删除行为
- 若已有 e2e 覆盖原始素材流，可补一个“打开弹窗并切换视频 tab”的场景

### 手动验证

- 打开素材管理页，点击“原始去字幕素材”，应出现居中大弹窗，而不是右侧抽屉
- 弹窗宽度视觉上约占浏览器的 2/3
- 每条素材卡都有“封面图 / 视频” tab
- 封面图 tab 默认展示封面
- 点击视频 tab 后，当前卡片可直接播放视频
- 删除素材后，列表与页面入口上的数量同步更新
- 空列表时空状态正确
- 小窗口下卡片会自动换列，预览比例仍正确

## 当前基线限制

在本次设计阶段的 worktree 基线检查中，相关测试状态如下：

- `tests/test_medias_raw_sources_translate.py -q -x`：通过（4 passed）

- `tests/test_medias_raw_sources_routes.py -q -x`：当前环境下运行超时，需在实现阶段单独确认是否为环境依赖或已有慢用例

结论：

- 当前已有测试并非全绿基线
- 后续实现时，需要把“本地 MySQL 未启动”和“routes 套件超时”视为既有环境限制单独标注，避免误判成新 UI 改造引入的问题

## 风险

1. 编辑页样式直接复用过多，可能影响现有编辑弹窗布局
2. 原始素材与编辑页素材的数据字段不完全一致，可能需要额外做前端字段兼容
3. 视频懒加载如果没有独立状态管理，容易重复请求或切 tab 闪烁
4. 固定 `180x320` 卡片后，若标题过长，卡片高度与按钮区排布需要重新收口

## 决策摘要

- 使用居中大弹窗替换右侧抽屉
- 使用每张卡片内部的“封面图 / 视频”双 tab，而不是全局 tab
- 预览框统一为 `180x320`、`9:16`
- 视频在卡片内直接播放
- 复刻编辑页的素材展现方式，但不在本次改造中扩展新的素材管理能力
