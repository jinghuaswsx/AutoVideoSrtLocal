# 后台可配置素材语种设计

**日期**: 2026-04-15  
**范围**: 系统设置页、素材语种配置、素材页语言读取逻辑  
**目标**: 把“产品素材 / 编辑素材”所用的小语种配置从后端写死改为后台可配置，并作为系统级配置供后续模块复用。

## 1. 背景

当前素材语种列表由后端固定返回，管理员无法在系统设置中直接调整语种集合、显示名称、启用状态和排序。  
这会带来两个问题：

- 语种调整需要改代码和发版，无法由运营或管理员自行维护。
- 后续如果其他模块也要复用这套语种，会继续复制同一份静态配置，增加冲突和不一致风险。

用户已确认：

- 该配置是系统级配置，不只服务于素材页。
- 后台需要支持新增、删除、启用/停用、排序、名称修改。
- 语言编码创建后不可修改。
- 本任务在独立分支中实施，避免影响主干上正在运行的其他任务。

## 2. 设计目标

- 以 `media_languages` 作为唯一语种配置源。
- 在后台“系统设置”页新增“素材语种配置”区域，支持前端可配置。
- 语言编码在创建后只读，不允许编辑。
- `en` 作为默认语种永久保留，不允许删除或停用。
- 已被素材、文案或主图使用的语种不允许删除，但允许停用。
- 停用语种不再出现在素材编辑页 tab 中，但历史数据保留。
- 素材页继续通过 `/medias/api/languages` 读取当前启用语种，避免前端重复维护配置。

## 3. 数据模型

继续使用现有表 `media_languages`：

- `code`: 语言编码，主键，创建后只读
- `name_zh`: 后台展示的中文名称
- `sort_order`: 排序值
- `enabled`: 是否启用

不新增第二份 JSON 或系统设置镜像，避免配置源分裂。

新增一个“使用情况统计”查询逻辑，按语言编码统计：

- `items_count`: `media_items` 中使用该语种的条数
- `copy_count`: `media_copywritings` 中使用该语种的条数
- `cover_count`: `media_product_covers` 中使用该语种的条数
- `in_use`: 任一统计大于 0 即为真

## 4. 后端方案

### 4.1 DAO / Settings 层

在 `appcore` 中新增或扩展一组语种管理方法：

- 列出全部语种（含停用语种）
- 列出启用语种（供素材页使用）
- 创建语种
- 更新语种名称 / 排序 / 启用状态
- 删除语种
- 查询语种使用情况

约束规则：

- 新建编码时统一转小写保存。
- 编码仅允许 `2-8` 位小写字母、数字或连字符，兼容未来如 `pt-br`。
- 创建时校验唯一性。
- 更新时不允许修改 `code`。
- `en` 不允许删除，也不允许停用。
- 删除前必须检查 `in_use`，已使用则返回错误。

### 4.2 管理接口

在 `web/routes/admin.py` 下新增管理员专用接口：

- `GET /admin/api/media-languages`
- `POST /admin/api/media-languages`
- `PUT /admin/api/media-languages/<code>`
- `DELETE /admin/api/media-languages/<code>`

接口返回统一包含：

- 基础字段：`code`、`name_zh`、`sort_order`、`enabled`
- 统计字段：`items_count`、`copy_count`、`cover_count`、`in_use`
- 必要时带 `error` 消息，供行内提示使用

### 4.3 素材页读取逻辑

`/medias/api/languages` 继续存在，但改为读取“当前启用语种”，按：

1. `sort_order` 升序
2. `code` 升序

这样系统设置页保存后，素材页无需额外改动即可自动反映新配置。

## 5. 前端方案

在现有 [admin_settings.html](G:\Code\AutoVideoSrt\.worktrees\codex-admin-media-languages-settings\web\templates\admin_settings.html) 中追加“素材语种配置”卡片，不拆新页面。

### 5.1 列表结构

列定义：

- 语言编码
- 语言名称
- 排序
- 启用
- 使用情况
- 操作

每行行为：

- 编码：新建时可编辑；已有记录显示为只读标签
- 名称：文本输入框
- 排序：数字输入框
- 启用：勾选框或开关
- 使用情况：展示“素材 / 文案 / 主图”计数
- 操作：保存、删除

### 5.2 交互规则

- 点击“新增语种”后，在列表底部插入一条可编辑空行。
- 新行允许填写编码、名称、排序和启用状态。
- 保存成功后刷新列表并显示顶部成功提示。
- 保存失败时，在当前行下显示错误信息，保留用户输入。
- 删除时：
  - 未使用语种：允许删除
  - 已使用语种：删除按钮禁用，并提示“已有关联数据，只能停用”
- `en` 行展示“默认语种”标识，并禁用删除和停用。

### 5.3 视觉风格

遵循现有 Ocean Blue Admin 风格：

- 使用现有 token，不引入紫色
- 卡片白底、浅边框、大圆角
- 状态提示使用已有 success / warning / danger 色
- 交互组件与当前系统设置页的输入框和按钮风格保持一致

## 6. 错误处理

后端错误场景：

- 编码为空或格式不合法
- 编码重复
- 尝试修改编码
- 尝试停用 `en`
- 尝试删除 `en`
- 尝试删除已被使用的语种

前端展示策略：

- 表单级成功提示继续沿用 flash 或轻量消息
- 行级失败显示在当前行下方，避免用户不知道哪一项出错

## 7. 测试策略

需要覆盖三类测试：

### 7.1 DAO / 设置层

- 创建语种成功
- 重复编码被拒绝
- 语言编码更新被拒绝
- `en` 禁删禁停用
- 已使用语种删除被拒绝
- 启用语种列表按排序返回

### 7.2 Admin 路由

- 管理员可获取全部语种列表
- 管理员可新增语种
- 管理员可更新名称 / 排序 / 启用状态
- 非法请求返回明确错误
- 删除受保护语种或在用语种时返回错误

### 7.3 页面回归

- 系统设置页渲染“素材语种配置”区域
- 素材页 `/medias/api/languages` 只返回启用语种
- 停用语种不会出现在编辑素材弹窗 tab 中

## 8. 风险与边界

- 本次不处理历史语种编码迁移，因为编码创建后不可改。
- 本次不扩展到文案翻译、文本翻译等其他模块的语言选择 UI，只把配置源做成系统级可复用。
- 若后续其他模块要复用，只需统一改为读取同一套语种配置接口或 DAO。

## 9. 实施结果预期

完成后，管理员可以在后台系统设置页直接维护素材语种：

- 删掉不想启用的语种
- 新增新语种
- 调整显示顺序
- 修改显示名称
- 在不丢历史数据的前提下停用已有语种

素材编辑页会自动显示最新启用语种集合，不再需要通过改代码来调整小语种列表。
