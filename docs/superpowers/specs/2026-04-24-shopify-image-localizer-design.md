# Shopify 图片本地化替换工具设计

> 状态：需求已确认
> 范围：本地桌面 EXE 工具、配套服务端只读接口、Shopify 后台双页面自动化
> 前置参考：`docs/superpowers/specs/2026-04-20-link-check-desktop-design.md`、`docs/superpowers/specs/2026-04-19-medias-localized-detail-image-translation-design.md`

## 1. 背景

当前团队已经能产出同一商品的多语种翻译图片，但把这些图片真正替换进 Shopify 后台仍然高度依赖人工重复操作。用户每处理一个商品，通常需要反复打开 `EZ Product Image` 和 `Translate and Adapt` 两个插件页面，搜索商品，确认语言，把本地翻译图与页面现有英文图进行配对，再逐张上传。商品语种一多，重复成本会快速放大。

本次要做的不是新的服务端管理页面，而是一个运行在用户本机上的 EXE 小工具。它属于仓库里的 `tools` 目录下的本地工具代码，打包后由用户在 Windows 电脑上直接运行。这个工具负责：

1. 接收 `product_code` 和目标语言。
2. 调用现有项目后端新增的支持接口，拉取 Shopify 商品 ID 以及待处理图片 URL。
3. 把图片下载到 EXE 同级目录。
4. 打开固定持久化目录的调试浏览器。
5. 自动进入 Shopify 的两个目标页面。
6. 利用页面里的英文图位和现有链接检查能力完成图片归类、配对和替换。

核心目标是把“重复但需要一定配对判断”的图片本地化工作尽可能自动化，同时保持错误替换的风险足够低。

## 2. 目标

### 2.1 用户目标

1. 在一个约 `800x800` 的本地窗口里输入 `product_code`，选择语言后点击一次 `开始替换图片`。
2. 程序先从服务端拿到该商品该语种的图片清单，并逐张下载到本地，不走压缩包流程。
3. 程序自动使用持久化浏览器目录 `C:\chrome-shopify-image` 打开 Shopify。
4. 如果 Shopify 登录态失效，程序暂停等待用户手动登录；只要登录恢复且输入条件已满足，程序自动继续，不要求额外点击“继续”。
5. 同一次任务里同时处理：
   - `EZ Product Image` 中的轮播图语言替换
   - `Translate and Adapt` 中商品详情编辑区的图片替换
6. 如果两条链路并发稳定，就并发执行；如果页面互相冲突，就自动退回串行，但对用户仍表现为“一次开始，两件都完成”。
7. 配对不确定的图片宁可跳过并标记为待人工复核，也不做高风险误替换。

### 2.2 系统目标

1. 工具源码落在仓库 `tools` 目录下，定位为本地桌面工具，而不是服务端软件。
2. 最大化复用现有 `link_check_desktop`、`appcore.link_check_runtime`、`appcore.link_check_compare` 一类能力，避免重新发明配对内核。
3. 服务端只补充支持本地工具调用的接口和数据查询逻辑，不把自动化执行放到服务端。
4. 本地目录、日志、截图、清单都要可复查，便于重跑和排错。

## 3. 非目标

本次明确不做以下内容：

1. 不把这个功能做成 Web 端页面，也不依赖用户在服务端机器上操作。
2. 不做压缩包下载与解压流程，接口只返回图片地址列表。
3. 不做商品批量队列处理，首版按单商品单语言运行。
4. 不做“只跑轮播图”或“只跑详情图”的用户模式切换，点击开始后默认两条链路都要执行。
5. 不要求服务端提前把图片分成“轮播图”和“详情图”两组，因为当前实际数据就是混在一起的。
6. 不把 Playwright 浏览器嵌进桌面 UI 里，浏览器保持独立窗口。
7. 不做自动发布、自动更新、云端任务调度或服务端回写执行日志。

## 4. 方案对比

### 方案 1：复用 `link_check_desktop` 骨架，新建本地 EXE 工具

技术形态采用 `Tkinter + Playwright + PyInstaller`。源码新建在 `tools` 目录下，复用现有桌面工具的窗口、配置、打包和图片匹配思路。

优点：

1. 最接近当前仓库已有实践，落地最快。
2. Playwright 对持久化登录态、显式等待、多窗口控制更稳。
3. 便于复用现有链接检查配对逻辑。

缺点：

1. 首版 UI 会偏朴素。
2. 需要处理 Playwright 打包和浏览器运行时依赖。

### 方案 2：重新做一个 `PySide6` 原生桌面工具

优点：

1. UI 可扩展性更强，适合以后做复杂任务工作台。
2. 更容易做精细化状态面板。

缺点：

1. 当前仓库没有现成主线，启动成本明显更高。
2. 对“尽快跑通真实业务”没有额外收益。

### 方案 3：继续做 Web Tools 页面，再配一个本地自动化助手

优点：

1. 服务端数据接入自然。
2. 网页入口统一。

缺点：

1. 会把一个本来应该本地运行的工具拆成两套系统。
2. 浏览器持久化目录、本地落盘和本机登录态都会变得更绕。
3. 不符合用户已经明确的“本地 EXE，小工具，放到 `tools` 目录”的定位。

结论：采用方案 1。

## 5. 总体边界与目录落位

### 5.1 本地工具边界

本次主交付物是一个本地 EXE 工具，源码放在仓库 `tools` 目录下，例如：

```text
tools/
  shopify_image_localizer/
    __init__.py
    main.py
    gui.py
    controller.py
    settings.py
    storage.py
    api_client.py
    downloader.py
    matcher.py
    browser/
      session.py
      ez_flow.py
      translate_flow.py
    packaging/
      shopify_image_localizer.spec
```

这个工具运行在用户自己的 Windows 电脑上，负责下载图片、打开本机持久化浏览器、执行页面自动化、记录本地日志和生成结果。

### 5.2 服务端支持边界

服务端只承担“给本地工具提供任务素材”的职责，不负责远程执行浏览器自动化。支持改动预计落在：

1. `web/routes/medias.py` 或同类 OpenAPI 路由文件：提供本地工具调用的新接口。
2. `appcore/medias.py` 及相关查询逻辑：根据 `product_code` 和目标语言返回 Shopify 商品 ID 与图片 URL 列表。
3. 如有需要，补充 `media_languages` 返回字段，供本地工具下拉框显示。

因此，这仍然是“本地工具 + 服务端支持接口”的模式，不是“服务端软件替用户跑 Shopify 浏览器”。

## 6. 用户界面设计

主窗口固定约 `800x800`，以任务操作与状态反馈为主，不追求复杂工作台。

### 6.1 顶部输入区

1. `产品 ID` 输入框：
   - 实际输入的是 `product_code`，不是数据库自增 ID，也不是 Shopify 商品 ID。
2. `目标语言` 下拉框：
   - 来源：`media_languages` 中 `enabled=1` 的语言。
   - 显示格式：`中文名（大写语言码/小写目录码）`。
   - 现有表如果只有 `code`，首版先按 `德语（DE/de）` 这种形式展示，也就是把同一个 `code` 分别做大写和小写显示；如果后续表里新增独立目录码字段，再切换为真实值。
3. `开始替换图片` 按钮：
   - 用户必须在填好 `product_code` 并选好语言后点击一次，任务才会启动。

### 6.2 状态区

窗口主体使用滚动日志和分区状态展示，至少包含：

1. 总状态：待输入、拉取数据、下载图片、等待登录、处理中、部分完成、失败。
2. 登录状态：是否检测到 Shopify 有效登录态。
3. 下载进度：总图片数、成功数、失败数。
4. `EZ Product Image` 进度。
5. `Translate and Adapt` 进度。
6. 结果摘要：成功、跳过、待人工复核、失败。
7. 最近日志滚动区。

## 7. 执行总流程

用户点击 `开始替换图片` 后，固定按以下顺序运行：

1. 校验 `product_code` 已填写。
2. 校验目标语言已选择。
3. 调用服务端新接口，拿到：
   - `shopify_product_id`
   - 目标语言信息
   - 英文参考图列表
   - 目标语言混合图片列表
4. 在 EXE 同级目录创建本次任务目录。
5. 先下载所有参考图和目标语言图到本地。
6. 启动使用 `C:\chrome-shopify-image` 的持久化 Playwright 浏览器实例。
7. 检测 Shopify 登录态：
   - 已登录则继续。
   - 未登录则停在浏览器，让用户手动登录。
   - 程序后台定时重试检测；只要恢复登录且输入条件已满足，就自动继续。
8. 在同一个浏览器实例里打开两个独立页面或窗口：
   - `EZ Product Image`
   - `Translate and Adapt`
9. 默认并发执行两个流程。
10. 如果检测到编辑冲突、跳页干扰、页面互相影响或长时间超时，则自动退回串行。
11. 输出最终状态、日志、截图与清单。

## 8. 服务端接口契约

### 8.1 入参

建议新增一个专门给本地工具调用的接口，例如：

```text
POST /openapi/medias/shopify-image-localizer/bootstrap
```

请求体：

```json
{
  "product_code": "dino-glider-launcher-toy-rjc",
  "lang": "de"
}
```

### 8.2 出参

接口应一次返回本地工具所需的全部关键数据，不要求客户端再拼多次接口。推荐返回：

```json
{
  "product": {
    "product_code": "dino-glider-launcher-toy-rjc",
    "shopify_product_id": "8552296546477",
    "name": "Dino Glider Launcher Toy"
  },
  "language": {
    "code": "de",
    "name_zh": "德语",
    "shop_locale": "de"
  },
  "reference_images": [
    {
      "id": "en-ref-1",
      "filename": "en_ref_1.jpg",
      "url": "https://..."
    }
  ],
  "localized_images": [
    {
      "id": "de-target-1",
      "filename": "de_target_1.jpg",
      "url": "https://..."
    }
  ]
}
```

关键约束：

1. 服务端目前不需要预先区分 `carousel` 和 `detail`。
2. 服务端返回的是：
   - 英文参考图集合
   - 当前目标语言的混合图集合
3. 本地工具负责在 Shopify 页面里反向判断这些图分别属于轮播图还是详情图。

### 8.3 接口失败约定

首版至少覆盖以下失败情形：

1. `400`：参数非法，如缺少 `product_code` 或 `lang`。
2. `404`：找不到 `product_code` 对应商品。
3. `409`：商品缺少 `shopify_product_id` 或目标语言不可用。
4. `409`：该语种没有可下载的本地化图片，或英文参考图缺失。
5. `500`：服务端内部错误。

## 9. 本地目录结构

下载目录位于 EXE 同级目录，产品文件夹名直接使用 `product_code`，其下再按语言划分：

```text
<EXE目录>/
  shopify_image_localizer.exe
  shopify_image_localizer_config.json
  dino-glider-launcher-toy-rjc/
    de/
      source/
        en/
        localized/
      classify/
        ez/
        taa/
      screenshots/
        ez/
        taa/
      manifest.json
      run.log
```

说明：

1. `source/en`：英文参考图下载结果。
2. `source/localized`：目标语种混合图下载结果。
3. `classify/ez`：判定属于轮播图的候选图。
4. `classify/taa`：判定属于详情图的候选图。
5. `screenshots/*`：页面证据截图，便于排错。
6. `manifest.json`：记录 URL、文件名、映射结果、Shopify 商品 ID、执行时间等。
7. `run.log`：记录逐步执行日志。

## 10. Shopify 页面直达策略

本工具不再依赖在插件页面里先搜索商品，而是直接打开用户已确认可用的后台地址：

### 10.1 `EZ Product Image`

```text
https://admin.shopify.com/store/0ixug9-pv/apps/ez-product-image-translate/product/<shopify_product_id>
```

说明：

1. 商品定位通过接口直接返回的 `shopify_product_id` 完成。
2. 页面打开后，如插件内部仍要求切换语言，则在该页面再选择目标语言。

### 10.2 `Translate and Adapt`

```text
https://admin.shopify.com/store/0ixug9-pv/apps/translate-and-adapt/localize/product?highlight=handle&id=<shopify_product_id>&shopLocale=<lang>
```

说明：

1. 该 URL 直接携带商品 ID 与目标语言代码。
2. 页面目标是定位商品翻译编辑页中左英文、右目标语种的编辑区，并替换其中的目标图片。

## 11. 登录态与浏览器策略

### 11.1 技术选型

采用 Playwright，而不是 Selenium，原因是：

1. 对持久化用户目录支持更直接。
2. 对多窗口、多页面控制更稳。
3. 等待页面状态和处理超时更可控。

### 11.2 浏览器运行规则

1. 浏览器用户目录固定为 `C:\chrome-shopify-image`。
2. 只启动一个持久化浏览器实例，不同时启动多个抢同一目录的浏览器进程。
3. 两条 Shopify 链路在同一个实例里开两个独立页面。

### 11.3 登录态规则

程序通过访问目标页来判断登录是否有效，出现以下任一情况就视为未登录或登录失效：

1. 被重定向到登录页、店铺选择页或权限页。
2. 页面长期拿不到目标插件内容区域。
3. 页面不断回到非目标 Shopify 后台入口。

缺少登录态时：

1. 程序进入 `等待登录` 状态。
2. 浏览器保持打开，用户手动完成登录。
3. 程序后台自动重新检测。
4. 只要登录态恢复，且此前已经满足 `product_code + language`，程序立刻继续。

## 12. 两条自动化链路

### 12.1 `EZ Product Image` 轮播图链路

流程如下：

1. 打开直达 URL。
2. 如果页面中还需要切语言，则选定目标语言。
3. 读取该页面当前商品对应的英文轮播图位。
4. 把页面英文图位与本地英文参考图做匹配，确认哪些参考图属于轮播图集合。
5. 再从目标语种混合图里找到与这些英文参考图对应的目标图。
6. 按页面要求执行添加、上传或替换。
7. 对成功项、跳过项、失败项分别记录。

### 12.2 `Translate and Adapt` 详情图链路

流程如下：

1. 打开直达 URL。
2. 定位商品翻译编辑页面中的左英文、右目标语种编辑区。
3. 在左侧英文内容里识别带图的内容块。
4. 把这些英文图片块与本地英文参考图做匹配，确认哪些参考图属于详情图集合。
5. 再从目标语种混合图里为这些详情图块找到对应目标图。
6. 在右侧目标语种编辑区执行图片替换。
7. 记录成功项、跳过项、失败项。

## 13. 混合图片归类与配对规则

由于服务端返回的本地化图片并未区分“轮播图”与“详情图”，归类必须由本地工具在 Shopify 实页完成。

### 13.1 归类原则

1. 先用 `EZ Product Image` 页面识别轮播图英文图位。
2. 轮播图链路命中的参考图及其目标图先占用。
3. 再用 `Translate and Adapt` 页面识别详情图英文块。
4. 详情图链路优先使用剩余未占用的目标图。

### 13.2 匹配原则

优先复用现有链接检查链路中的图像比较能力，推荐顺序：

1. 参考图最佳匹配。
2. 二值快检。
3. 同图判断。
4. 必要时才使用更重的判定逻辑。

### 13.3 安全规则

1. 只有高置信度配对才自动替换。
2. 低置信度配对标记为 `需人工复核`，本次跳过。
3. 同一张目标语种图片默认只能分配给一个页面链路。
4. 如果两条链路都强匹配到同一张目标图，则标记为冲突并跳过，不强行替换。

## 14. 并发与回退策略

用户已经明确一次任务里两件事情都要做，因此工具内部采用：

1. 默认并发：两个窗口各跑一条链路。
2. 自动回退：只要检测到以下问题之一，就退回串行继续：
   - 页面互相触发跳转
   - Shopify 后台编辑态冲突
   - 插件页面刷新后丢失上下文
   - 上传动作反复超时且怀疑被另一页面影响

对用户而言，无需理解内部模式切换；界面只需显示“并行处理中”或“串行恢复中”。

## 15. 重试、失败与结果判定

### 15.1 重试策略

1. 服务端接口请求：失败后重试 2 次。
2. 单张图片下载：失败后重试 1 次。
3. Shopify 页面打开：失败后重试 2 次。
4. 页面元素定位：允许刷新后再尝试 1 次。
5. 单张上传动作：失败后重试 1 次。

### 15.2 登录失效

运行过程中如果登录失效，不直接把任务判死，而是回到 `等待登录`，待登录恢复后继续。

### 15.3 最终状态

1. `完成`：轮播图链路和详情图链路都成功结束，且没有未处理冲突项。
2. `部分完成`：至少一条链路成功，但存在失败、跳过或人工复核项。
3. `失败`：两条链路都未完成，或者前置数据/登录/页面流程无法继续。

## 16. 日志、证据与结果展示

窗口里至少展示：

1. 当前阶段。
2. 登录状态。
3. 下载进度。
4. `EZ Product Image` 进度。
5. `Translate and Adapt` 进度。
6. 最终摘要。

本地落盘至少保留：

1. `manifest.json`：输入参数、接口返回、图片清单、分类结果、配对结果、最终状态。
2. `run.log`：文本日志。
3. `screenshots/ez/*`、`screenshots/taa/*`：关键失败与关键成功证据截图。

这样即使首版没有复杂结果 UI，用户也能通过本地文件快速定位问题。

## 17. 验收标准

首版验收至少覆盖以下场景：

### 17.1 基本成功链路

使用以下样例数据：

```text
product_code: dino-glider-launcher-toy-rjc
shopify_product_id: 8552296546477
lang: de
```

验收期望：

1. 能从接口拿到商品信息和图片地址列表。
2. 能在 EXE 同级目录正确落盘为 `<product_code>/<lang>/...`。
3. 能使用 `C:\chrome-shopify-image` 成功启动持久化浏览器。
4. 若已登录，程序可直接进入两个目标 Shopify 页面。
5. 能完成至少一轮真实的图片归类、配对和上传动作。

### 17.2 登录恢复链路

1. 在浏览器未登录时点击开始。
2. 程序应停在等待登录状态，而不是直接报错退出。
3. 用户手动登录后，程序应自动恢复执行。

### 17.3 不确定配对链路

1. 当目标图无法高置信度配对时，程序不应误上传。
2. 该项应被记录为 `需人工复核`。

### 17.4 并发回退链路

1. 两页面并发执行时若发生干扰，程序应自动切换为串行模式继续完成。
2. 这个切换过程不需要用户额外操作。

## 18. 风险与约束

1. `EZ Product Image` 与 `Translate and Adapt` 都属于第三方或外部后台页面，DOM 结构可能变化，自动化选择器必须预留调整空间。
2. `media_languages` 当前若没有独立的“语言子目录编码”字段，首版 UI 只能用 `code` 的大小写形式代替。
3. `shopify_product_id` 必须能由服务端可靠返回；本地工具不应自己去猜 Shopify 商品 ID。
4. 首版的关键成功标准不是“所有商品零人工介入”，而是“绝大多数重复劳动可自动完成，并且不做高风险误替换”。

## 19. 结论

本次设计采用“本地 EXE 工具放在 `tools` 目录 + 服务端补支持接口 + Playwright 持久化浏览器自动化”的组合方案。点击一次 `开始替换图片` 后，工具先下载图片，再校验登录，接着同时处理 Shopify 的轮播图和详情图替换。服务端只负责给出图片与商品元数据，本地工具负责真正的页面归类、配对、上传和证据留存。

这份设计的重点不是做一个新的后台系统，而是把用户现在最重复、最耗时、最容易出错的 Shopify 图片本地化操作，收敛成一个可反复执行的本地小工具。
